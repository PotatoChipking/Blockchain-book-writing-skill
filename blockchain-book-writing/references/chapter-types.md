# Chapter Types

Use these templates as defaults. Adapt them to the user's outline and book context instead of forcing every chapter into the same shape.

## Technical Principle

For consensus, virtual machines, account models, state models, Merkle trees, cross-chain systems, privacy computing, zero-knowledge proofs, and data availability.

Default structure:

1. 问题背景
2. 基本概念
3. 核心原理
4. 执行流程
5. 工程实现要点
6. 性能、安全性或可扩展性分析
7. 典型应用场景
8. 局限与发展方向
9. 小结

Requirements:

- Explain why the mechanism is needed, not only what it is.
- Break complex mechanisms into flows.
- Ground engineering discussion in modules, states, data flow, or call chains.
- For algorithms or protocols, explain inputs, processing, and outputs.

## Architecture Design

For consortium-chain platforms, financial infrastructure, trusted data circulation, payment networks, storage architecture, and cross-chain architecture.

Default structure:

1. 架构设计背景
2. 系统目标与约束
3. 总体架构
4. 核心模块
5. 关键流程
6. 数据流与状态变化
7. 安全、性能与可扩展性设计
8. 工程实现挑战
9. 小结

Requirements:

- Explain why the architecture is designed this way.
- Do not only list module names.
- Explain module interaction, key data flow, tradeoffs, and implementation constraints.

## Project Case

For projects such as Arc, Tempo, Plasma, Canton Network, Provenance, Fabric, Corda, Quorum, Aptos, Diem, and Hyperledger.

Default structure:

1. 项目出现背景
2. 项目定位
3. 核心架构
4. 关键技术特点
5. 典型应用场景
6. 与传统方案的差异
7. 面临的挑战与不确定性
8. 对联盟链或金融基础设施的启示
9. 小结

Requirements:

- Do not write product promotion.
- Do not present project vision as implemented reality.
- Explain why the project emerged at that time.
- Connect technical route with application scenarios.
- Return to the book's theme, such as consortium chains, financial infrastructure, stablecoin payments, RWA, or trusted data circulation.

## Comparison

For comparing blockchain models, technical routes, projects, architectures, and application modes.

Default structure:

1. 对比背景
2. 对比对象定义
3. 对比维度
4. 技术路线差异
5. 工程实现差异
6. 治理与合规差异
7. 适用场景差异
8. 优势与局限总结
9. 趋势判断

Common dimensions:

- 准入机制
- 共识机制
- 交易执行模型
- 状态管理方式
- 隐私保护能力
- 性能与最终性
- 治理模式
- 合规适配
- 开发者生态
- 机构采用难度
- 运维复杂度
- 可扩展性
- 成本结构

Requirements:

- Avoid simple "A is better than B" claims.
- Tie conclusions to specific constraints and dimensions.
- Avoid absolute judgments.

## Trend Review

For industry trends, technical evolution, future directions, and financial infrastructure change.

Default structure:

1. 发展背景
2. 核心驱动因素
3. 主要技术路线
4. 代表性案例
5. 对产业或工程实践的影响
6. 风险与挑战
7. 未来趋势
8. 小结

Requirements:

- Explain technical, policy, market, or industrial drivers.
- Distinguish short-term hype from long-term structural change.
- Keep future judgments cautious.

## Engineering Practice

For deployment, testing, performance optimization, operations, security hardening, on-chain/off-chain collaboration, storage optimization, and state verification.

Default structure:

1. 工程背景
2. 实际问题
3. 设计目标
4. 实现方案
5. 关键流程
6. 测试与验证
7. 性能或可靠性分析
8. 常见问题与优化方向
9. 小结

Requirements:

- Show real engineering constraints.
- Explain why a solution is chosen.
- Include understandable flows and module relationships.
- Do not invent performance metrics.

## Historical Evolution

For Libra/Diem, consortium-chain history, public-chain evolution, stablecoin infrastructure development, and enterprise-blockchain rise and decline.

Default structure:

1. 起点与背景
2. 关键阶段
3. 技术路线变化
4. 代表性事件或项目
5. 成功经验与失败原因
6. 对当前技术路线的影响
7. 未来启示
8. 小结

Requirements:

- Keep timelines clear.
- Treat causality cautiously.
- Do not reduce complex failures to one cause.
- Analyze technical, commercial, regulatory, and ecosystem factors together.

## Chapter Continuity

When writing consecutive chapters, each opening should connect to the parent topic and explain why the section appears. Each ending should return to the book's main line and create a natural transition.

For a group of case chapters, use similar analysis dimensions such as background, positioning, architecture, scenarios, relation to consortium chains or public chains, limits, and financial-infrastructure implications. Avoid identical openings and sentence patterns.
